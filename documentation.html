Getting started / Introduction

Introduction

QuickForge is a production-oriented scaffolding engine for web applications. It generates fully configured, deployment-ready projects from opinionated templates — with Auth, database, styling, and CI/CD pre-wired.

What is QuickForge

Most scaffolding tools give you a project skeleton. QuickForge gives you a working application. Every template starts from a state where npm run dev works on the first try, npm run build produces a deployable artefact, and the CI pipeline passes before you write a single line of business logic.

The engine pulls templates from a remote registry at scaffold time, so you always get the latest version of each dependency and configuration pattern rather than a stale snapshot baked into an npm package.

QuickForge requires Node.js 18+ and Bun 1.0+. If you prefer npm or pnpm, pass --pm npm or --pm pnpm to any command.

Design philosophy

QuickForge is built on three principles:

Opinionated by default, configurable when it matters

The defaults are chosen to reflect production best practices — TypeScript strict mode, ESLint flat config, Prettier, Husky pre-commit hooks. You opt out of defaults rather than opt in to them.

No abstraction on top of your stack

QuickForge writes standard Next.js, React, Hono, or SvelteKit code. There is no QuickForge runtime, no QuickForge component library, and no QuickForge-specific APIs. The generated project is a plain Next.js project the day after scaffolding.

Reproducible across machines and time

Lockfiles are committed, Docker builds are deterministic, and CI uses frozen installs. A project scaffolded today should build identically six months from now on a different machine.

How it works

Running npx quickforge create fetches the selected template from the QuickForge registry over HTTPS, writes the interpolated source to disk, installs dependencies with your chosen package manager, initialises a Git repository, and runs the post-scaffold validation suite to confirm the project builds cleanly.

Getting started / Installation

Installation

System requirements and install methods. QuickForge can be run with npx for one-off scaffolding, or installed globally for frequent use.

System requirements

RequirementMinimum versionNotes
Node.js18.17.0LTS or Current. Used for npx and package scripts.
Bun1.0.0Default package manager and task runner in generated projects.
Git2.30.0Required for repository initialisation and Husky hooks.
Operating system—macOS 12+, Linux (glibc 2.28+), Windows WSL2

Using npx (recommended)

The simplest way to use QuickForge is via npx. No global installation is required — the CLI is fetched on demand and cached locally.

bash
npx quickforge create my-app
No global install is required when using npx. npm will always resolve the latest stable version of QuickForge on each invocation.

Global install

If you scaffold projects regularly, a global install avoids the npx lookup on every invocation.

bash
# npm npm install -g quickforge # Bun bun add -g quickforge # pnpm pnpm add -g quickforge

Verify installation

After installing, confirm the CLI is reachable and check the version:

bash
quickforge --version # quickforge/2.4.0 linux-x64 node-v22.3.0

Updating

To update a globally installed QuickForge to the latest stable release:

bash
# npm npm update -g quickforge # Bun bun update -g quickforge

Alternatively, use the built-in upgrade command which also handles project file synchronisation: quickforge upgrade. See the quickforge upgrade reference for details.

Getting started / Quickstart

Quickstart

Scaffold and run a production-ready Next.js 15 app in under 2 minutes. This guide walks from zero to a running dev server with authentication and database connected.

Step 1 — Scaffold

Run the create command with your desired stack and features. QuickForge will pull the template from the registry, interpolate your project name, install dependencies with Bun, and initialise a Git repository.

bash
npx quickforge create my-app \ --stack next-js-v15 \ --auth github \ --db postgresql

Step 2 — Environment

Copy the example environment file and fill in the required credentials. Every variable is validated at startup by the t3-env schema in src/lib/env.ts — the server will refuse to start if a required variable is missing.

bash
cd my-app cp .env.example .env.local

Edit .env.local and set the following values:

VariableHow to get it
AUTH_SECRETGenerate with openssl rand -hex 32
DATABASE_URLPostgreSQL connection string, e.g. from Neon or a local Docker container
GITHUB_CLIENT_IDGitHub OAuth App → Settings → Developer settings
GITHUB_CLIENT_SECRETSame OAuth App page as above

Step 3 — Database

Push the Prisma schema to your database. This creates the tables for the User, Account, and Session models that Auth.js requires.

bash
bun run db:push
db:push calls prisma db push and is intended for development. In production CI, use bun run db:migrate which calls prisma migrate deploy to apply versioned migrations safely.

Step 4 — Run

Start the Next.js development server. Turbopack is enabled by default in the scaffold, giving significantly faster hot module replacement than webpack.

bash
bun run dev # ▲ Next.js 15.3.0 (Turbopack) # - Local: http://localhost:3000 # - Network: http://192.168.1.10:3000

Available scripts

ScriptDescription
bun run devStart Next.js dev server with Turbopack
bun run buildProduction build with standalone output
bun run startStart the compiled production server
bun run db:pushPush schema changes to database (dev)
bun run db:migrateApply versioned migrations (production)
bun run db:studioOpen Prisma Studio GUI at localhost:5555
bun run lintRun ESLint on the src/ directory
bun run typecheckRun tsc --noEmit for type checking
bun run testRun Vitest unit tests

Getting started / Project structure

Project structure

Layout conventions shared across all QuickForge templates. Understanding the directory layout makes it easier to extend your project and transfer knowledge between stacks.

Next.js 15 layout

The Next.js scaffold uses the App Router with a src/ directory for all application code.

tree
my-app/ ├── .github/ │ └── workflows/ │ └── ci.yml # Lint, typecheck, test, build ├── prisma/ │ ├── schema.prisma # Database schema │ └── migrations/ # Versioned migration files ├── src/ │ ├── app/ # App Router — pages and layouts │ │ ├── (auth)/ # Route group — auth-gated pages │ │ │ └── dashboard/ │ │ │ └── page.tsx │ │ ├── (public)/ # Route group — public pages │ │ │ └── page.tsx │ │ ├── api/ │ │ │ └── auth/ │ │ │ └── [...nextauth]/ │ │ │ └── route.ts # Auth.js route handler │ │ ├── layout.tsx # Root layout with providers │ │ └── globals.css # Tailwind CSS v4 entry point │ ├── components/ │ │ ├── ui/ # Primitive UI components │ │ └── layout/ # Header, footer, shell │ ├── lib/ │ │ ├── auth.ts # Auth.js configuration │ │ ├── db.ts # Prisma client singleton │ │ └── env.ts # t3-env typed environment schema │ └── middleware.ts # Auth session middleware ├── .env.example # Committed — no secrets ├── Dockerfile # Multi-stage production build ├── docker-compose.yml # Local dev with Postgres + Redis ├── next.config.ts # Turbopack, image domains, etc. └── tsconfig.json

Naming conventions

ConventionExampleReason
Route groups use parentheses(auth)/dashboardGroups related routes without affecting the URL path
Server Components by defaultpage.tsx has no directiveMaximises server-side rendering and reduces client bundle
Client Components marked explicitly'use client' at top of fileMakes the client boundary visible at a glance
lib/ for singletonslib/db.ts, lib/auth.tsSingle import location prevents multiple instances of Prisma
PascalCase for componentsUserAvatar.tsxDistinguishes component files from utility modules

Import paths

All templates configure TypeScript path aliases so imports are absolute and portable regardless of nesting depth.

json
// tsconfig.json { "compilerOptions": { "baseUrl": ".", "paths": { "@/*": ["./src/*"] } } } // Usage in any file import { db } from "@/lib/db"; import { UserAvatar } from "@/components/ui/UserAvatar";
All templates use the same src/ layout and the @/* alias, so patterns you learn in the Next.js template transfer directly to the SvelteKit and Node.js templates.

CLI reference / quickforge create

quickforge create

Scaffold a new project from a QuickForge template. The create command fetches the chosen template from the registry, resolves all dependencies, and writes the project to disk.

Syntax

bash
npx quickforge create [project-name] [flags]

Flags

FlagDefaultDescription
--stacknext-js-v15Template to scaffold. One of next-js-v15, react-spa, node-express, sveltekit.
--authnoneComma-separated OAuth providers. e.g. github,google,discord.
--dbnoneDatabase type. One of postgresql, mysql, sqlite, mongodb.
--ormprismaORM to use. One of prisma, drizzle.
--pmbunPackage manager. One of bun, pnpm, npm.
--featuresnoneAdditional integrations. Comma-separated: stripe, email, analytics.
--monorepofalseScaffold a Turborepo monorepo instead of a single app.
--no-gitfalseSkip git initialisation.

Examples

bash
# Interactive mode — launches the wizard npx quickforge create # Minimal scaffold — no auth, no database npx quickforge create my-site --stack next-js-v15 # Full-stack with PostgreSQL and GitHub OAuth npx quickforge create my-saas \ --stack next-js-v15 \ --auth github \ --db postgresql \ --features stripe,email # SvelteKit with SQLite for a lightweight project npx quickforge create my-blog \ --stack sveltekit \ --db sqlite \ --pm pnpm # Node.js API with Drizzle ORM, skipping git init npx quickforge create my-api \ --stack node-express \ --db postgresql \ --orm drizzle \ --no-git

quickforge add
CLI reference / quickforge add

quickforge add

Add an integration or feature to an existing QuickForge project without re-scaffolding. The add command detects your project stack, installs the required packages, writes integration files, and updates environment variable templates.

Syntax

bash
quickforge add [integration] [flags]

Available integrations

IntegrationWhat it adds
stripeStripe billing with webhook handler, customer portal route, and typed price ID helpers
emailResend API client and React Email templates with a pre-built welcome email
analyticsVercel Analytics and Speed Insights components injected into the root layout
s3AWS S3 / Cloudflare R2 file upload with presigned URL handler and client hook
redisUpstash Redis client with rate-limiting middleware pre-configured
sentrySentry error tracking with source map upload and Next.js instrumentation hook

Examples

bash
# Add Stripe billing quickforge add stripe # Add email with Resend provider (default) quickforge add email --provider resend # Add Redis rate limiting with Upstash quickforge add redis --provider upstash # Add Sentry with org and project flags quickforge add sentry --org my-org --project my-app

What add does

1
Detect project stack
Reads package.json to determine whether the project is Next.js, React SPA, Node.js, or SvelteKit. Selects the correct integration variant accordingly.
2
Install required packages
Calls the project's package manager (bun, npm, or pnpm) to install the integration's dependencies and dev-dependencies.
3
Write integration files
Creates source files under src/lib/ and src/app/api/ as appropriate. Existing files are never overwritten without a confirmation prompt.
4
Append env vars to .env.example
Appends required environment variable keys (without values) to .env.example and to the t3-env schema in src/lib/env.ts.
5
Update TypeScript types
Runs tsc --noEmit to confirm the integration compiles cleanly. Reports any type errors before finishing.
Run quickforge add from the project root — the directory that contains package.json. Running it from a subdirectory will cause stack detection to fail.

quickforge upgrade
CLI reference / quickforge upgrade

quickforge upgrade

Upgrade the QuickForge CLI and optionally sync a project's generated files with the latest template version. The sync feature diffs your scaffold files against the current template and presents a patch preview before applying.

Upgrade the CLI

Running quickforge upgrade without flags upgrades the CLI itself to the latest stable version, whether installed globally or as a local dev-dependency.

bash
quickforge upgrade # Checking registry for updates... # quickforge 2.3.1 → 2.4.0 # ✓ Upgraded successfully

Sync project files

The --sync flag compares your project's generated scaffold files against the latest template version in the registry. It shows a unified diff of what would change and prompts for confirmation before applying.

bash
quickforge upgrade --sync # Fetching latest next-js-v15 template (v3.1.0)... # Comparing against scaffold version v3.0.2... # # Modified: next.config.ts # Modified: .github/workflows/ci.yml # Modified: Dockerfile # # Apply changes? [y/N]

Flags

FlagDescription
--syncSync project scaffold files with the latest template version in the registry.
--dry-runShow what would change without writing any files. Use with --sync.
--forceApply sync changes without the confirmation prompt. Use in CI after reviewing a --dry-run.
--sync modifies files in your working directory. Commit or stash all local changes before running it so you can easily review or revert the applied patch.

What gets synced

The sync operation only touches files that were generated by QuickForge and are expected to be maintained by the template, not by you. It never modifies your application code.

File categoryGets synced?
Config files (next.config.ts, tsconfig.json, eslint.config.mjs)Yes
CI workflows (.github/workflows/)Yes
Docker files (Dockerfile, docker-compose.yml)Yes
App code (src/app/, src/components/)No
Database schema (prisma/schema.prisma)No
Environment files (.env.local, .env.example)No

Templates / Node.js API

Node.js API template

Lightweight, type-safe REST and RPC API using Hono v4 and tRPC v11, with Drizzle ORM and Zod validation. Optimised for edge deployment and low cold-start times.

OpenAPI spec generation via @hono/zod-openapi. The /docs route serves a Scalar UI.

Upstash Redis (optional, via --features redis) for rate limiting and caching.

Vitest for unit and integration tests with a lightweight in-memory test database.

GitHub Actions CI running tests and a Docker build on every push.

typescript
// src/routes/login/+page.server.ts import { fail, redirect } from "@sveltejs/kit"; import { lucia } from "$lib/server/auth"; import { db } from "$lib/server/db"; export const actions = { default: async ({ request, cookies }) => { const formData = await request.formData(); const email = formData.get("email") as string; const user = await db.query.users.findFirst({ where: (u, { eq }) => eq(u.email, email), }); if (!user) return fail(400, { message: "Invalid credentials" }); const session = await lucia.createSession(user.id, {}); const sessionCookie = lucia.createSessionCookie(session.id); cookies.set(sessionCookie.name, sessionCookie.value, { path: ".", ...sessionCookie.attributes, }); redirect(302, "/dashboard"); }, };

Configuration / quickforge.config.ts

quickforge.config.ts

Optional project-level configuration file. Placed at the project root, it customises scaffold behaviour, sets default flags, and loads plugins — so you don't have to repeat long CLI flags on every invocation.

Creating the config

Generate a fully commented config file with the init config command, or create it manually at the root of your project:

bash
# Generate from template quickforge init config # Or create manually at ./quickforge.config.ts

Full example

typescript

Using config with CLI

CLI flags always take precedence over config file values. The config file sets project-wide defaults; flags let you override individual options for a specific invocation without editing the file.

The config file is completely optional. Every option it exposes can be passed directly as a CLI flag to quickforge create. The file is most useful in team environments where you want consistent defaults across all developers.

Configuration / Environment variables

Environment variables

QuickForge uses t3-env to validate and type all environment variables at build time, preventing runtime crashes from missing or mistyped configuration values.

How it works

The src/lib/env.ts file defines a Zod schema for every environment variable your application needs. On startup — both in development and in the Next.js build step — t3-env validates all variables against the schema. If a required variable is missing or fails validation, the process exits with a descriptive error rather than crashing silently at the first runtime access.

typescript
// src/lib/env.ts import { createEnv } from "@t3-oss/env-nextjs"; import { z } from "zod"; export const env = createEnv({ server: { AUTH_SECRET: z.string().min(32), DATABASE_URL: z.string().url(), GITHUB_CLIENT_ID: z.string().optional(), GITHUB_CLIENT_SECRET: z.string().optional(), }, client: { NEXT_PUBLIC_APP_URL: z.string().url(), }, runtimeEnv: { AUTH_SECRET: process.env.AUTH_SECRET, DATABASE_URL: process.env.DATABASE_URL, GITHUB_CLIENT_ID: process.env.GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET: process.env.GITHUB_CLIENT_SECRET, NEXT_PUBLIC_APP_URL: process.env.NEXT_PUBLIC_APP_URL, }, });

Server vs client variables

Next.js enforces a strict boundary between server and client environment variables. Variables in the server block are never sent to the browser. Variables in the client block must be prefixed with NEXT_PUBLIC_ to be available in Client Components.

Adding a new variable

1
Add to .env.local
Add the variable with its value to your local .env.local file. This file is gitignored and never committed.
2
Add to .env.example
Add the key without a value to .env.example with a comment describing where to get the value. This file is committed.
3
Add to env.ts schema
Add the variable to the appropriate server or client block with a Zod validator, and add it to the runtimeEnv map.
4
Use via env import
Import and use the typed value: import { env } from "@/lib/env"; env.MY_VAR

Complete .env.example

bash
# ── Application ─────────────────────────────────────────────── NEXT_PUBLIC_APP_URL=http://localhost:3000 # ── Auth.js ─────────────────────────────────────────────────── # Generate: openssl rand -hex 32 AUTH_SECRET= # Required in production only NEXTAUTH_URL=https://myapp.com # ── Database ────────────────────────────────────────────────── # Format: postgresql://USER:PASSWORD@HOST:PORT/DATABASE DATABASE_URL= # ── GitHub OAuth ────────────────────────────────────────────── # Create at: github.com/settings/applications/new GITHUB_CLIENT_ID= GITHUB_CLIENT_SECRET= # ── Google OAuth ────────────────────────────────────────────── # Create at: console.cloud.google.com GOOGLE_CLIENT_ID= GOOGLE_CLIENT_SECRET=

CI/CD secrets

For GitHub Actions, add secrets at Settings → Secrets and variables → Actions. Reference them in workflows with ${{ secrets.DATABASE_URL }}.

For Vercel, set variables in the project dashboard under Settings → Environment Variables. Vercel injects them at build time and runtime automatically.

For Railway, use railway variables set KEY=value from the CLI or the Railway dashboard Variables tab.

Never commit .env.local or any file containing real secrets. Only .env.example — which contains no values — should be tracked by Git. The scaffold includes a .gitignore entry for all .env*.local files.

Plugins

QuickForge plugins extend the scaffold engine with custom integrations, file generators, and post-scaffold hooks. Install a plugin package, register it in your config, and it runs as part of the scaffold pipeline.

Installing a plugin

Plugins are npm packages. Install the package, then add it to the plugins array in your quickforge.config.ts:

bash
npm install quickforge-plugin-shadcn
typescript
// quickforge.config.ts export default defineConfig({ plugins: ["quickforge-plugin-shadcn"], });

Writing a plugin

A QuickForge plugin is a Node.js module that exports an object conforming to the QuickForgePlugin interface:

typescript
// Plugin interface (from "quickforge/plugin") export interface QuickForgePlugin { name: string; onAfterScaffold?(ctx: ScaffoldContext): Promise<void>; onFileWrite?(file: FileWriteEvent): Promise<FileWriteEvent>; onDependencyInstall?(deps: string[]): Promise<string[]>; } // Minimal example plugin that writes a CHANGELOG.md import type { QuickForgePlugin } from "quickforge/plugin"; import { writeFile } from "node:fs/promises"; import { join } from "node:path"; const changelogPlugin: QuickForgePlugin = { name: "changelog", async onAfterScaffold(ctx) { await writeFile( join(ctx.outputDir, "CHANGELOG.md"), `# Changelog\n\n## 0.1.0\n\n- Initial scaffold\n` ); }, }; export default changelogPlugin;

Deploy to Vercel

All QuickForge templates are optimised for Vercel deployment. The generated project includes a vercel.json and a GitHub Actions workflow that runs tests and deploys on every push to main.

One-command deploy

Install the Vercel CLI and deploy from the project root. The vercel.json in the scaffold configures the framework, region, and headers automatically.

bash
# Install Vercel CLI bun add -g vercel # Preview deployment vercel # Production deployment vercel --prod

Connecting GitHub

1
Push to GitHub
Push your scaffolded project to a new GitHub repository.
2
Import on vercel.com
Go to vercel.com/new, select Import Git Repository, and choose your repository. Vercel auto-detects Next.js and configures the build settings.
3
Set environment variables
In the Vercel project settings under Environment Variables, add all keys from your .env.example with their production values.
4
Automatic deploys
Every push to main triggers a production deploy. Pull requests get a preview deployment with a unique URL for review.

The included vercel.json

json
{ "framework": "nextjs", "regions": ["iad1"], "headers": [ { "source": "/(.*)", "headers": [ { "key": "X-Frame-Options", "value": "SAMEORIGIN" }, { "key": "X-Content-Type-Options", "value": "nosniff" }, { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" }, { "key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()" } ] } ] }

Railway
Deployment / Railway

Deploy to Railway

Railway is an excellent fit for QuickForge Node.js API and full-stack templates. It provisions PostgreSQL and Redis alongside your service automatically, with no infrastructure configuration required.

Deploy via CLI

Install the Railway CLI and deploy from your project directory. Railway detects the Dockerfile in the scaffold and uses it for every subsequent deploy.

bash
# Install Railway CLI bun add -g @railway/cli # Log in railway login # Initialise project (run once) railway init # Deploy railway up

Provision a database

Add a managed PostgreSQL instance to your Railway project. Railway automatically injects DATABASE_URL into your service's environment — no manual copy-paste required.

bash
# Add PostgreSQL railway add --database postgresql # Add Redis (optional) railway add --database redis

Environment variables

Set application secrets from the CLI. Railway stores them encrypted and makes them available at runtime:

bash
railway variables set AUTH_SECRET=$(openssl rand -hex 32) railway variables set JWT_SECRET=$(openssl rand -hex 32) railway variables set NODE_ENV=production

The Dockerfile

Railway detects the Dockerfile in your project root and uses it for builds. The scaffold's multi-stage Dockerfile produces a minimal production image — no additional Railway configuration is needed.

Healthcheck

The Node.js API template includes a /health endpoint that Railway uses to determine whether a deploy succeeded before routing traffic:

typescript
// src/routers/health.ts health.get("/", (c) => c.json({ status: "ok", uptime: process.uptime() }) );
Railway supports private networking between services in the same project. The Node.js API template checks for a RAILWAY_PRIVATE_DOMAIN environment variable at startup and switches its database connection to the private network address when present, eliminating egress costs.

Deployment / Docker

Docker

Every QuickForge template ships with a production-optimised multi-stage Dockerfile and a docker-compose.yml for local development with Postgres and Redis.

The generated Dockerfile

The scaffold produces a three-stage Dockerfile: deps installs production dependencies, builder runs the build, and runner copies only the compiled output into a minimal Alpine image.

dockerfile
# ── Stage 1: install dependencies ───────────────────────────── FROM node:22-alpine AS deps RUN apk add --no-cache libc6-compat WORKDIR /app COPY package.json bun.lockb ./ RUN npm install -g bun && bun install --frozen-lockfile # ── Stage 2: build ──────────────────────────────────────────── FROM node:22-alpine AS builder WORKDIR /app COPY --from=deps /app/node_modules ./node_modules COPY . . ENV NEXT_TELEMETRY_DISABLED=1 RUN npm install -g bun && bun run build # ── Stage 3: production runner ──────────────────────────────── FROM node:22-alpine AS runner WORKDIR /app ENV NODE_ENV=production ENV NEXT_TELEMETRY_DISABLED=1 # Create non-root user RUN addgroup --system --gid 1001 nodejs RUN adduser --system --uid 1001 nextjs COPY --from=builder /app/public ./public COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./ COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static USER nextjs EXPOSE 3000 ENV PORT=3000 CMD ["node", "server.js"]

Build and run

bash
# Build the image docker build -t my-app . # Run with your local env file docker run -p 3000:3000 --env-file .env.local my-app

docker-compose for local dev

The scaffold includes a docker-compose.yml that starts your application alongside a PostgreSQL database and Redis instance:

yaml
services: app: build: . ports: - "3000:3000" env_file: .env.local depends_on: postgres: condition: service_healthy redis: condition: service_healthy postgres: image: postgres:16-alpine environment: POSTGRES_USER: postgres POSTGRES_PASSWORD: postgres POSTGRES_DB: myapp ports: - "5432:5432" volumes: - postgres-data:/var/lib/postgresql/data healthcheck: test: ["CMD-SHELL", "pg_isready -U postgres"] interval: 5s timeout: 5s retries: 5 redis: image: redis:7-alpine ports: - "6379:6379" healthcheck: test: ["CMD", "redis-cli", "ping"] interval: 5s timeout: 5s retries: 5 volumes: postgres-data:

Build args

ArgDefaultDescription
NODE_ENVproductionNode.js environment mode
NEXT_TELEMETRY_DISABLED1Disables Next.js telemetry in CI and Docker builds

Multi-platform builds

To build a single image for both AMD64 (Linux servers) and ARM64 (Apple Silicon, Graviton) use Docker Buildx:

bash
docker buildx build \ --platform linux/amd64,linux/arm64 \ --tag my-app:latest \ --push \ .
